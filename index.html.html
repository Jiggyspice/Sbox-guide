Chapter 01

What Is S&Box?

S&Box is a game creation platform built on Valve's Source 2 engine — the same technology behind Half-Life: Alyx, Counter-Strike 2, and Deadlock. Created by Facepunch (the studio behind Garry's Mod and Rust), it gives you access to a professional-grade engine with C# scripting, a modern editor, and built-in multiplayer.

Think of it as a next-generation successor to Garry's Mod, but designed for creating standalone games — not just mods. Since March 2026, Facepunch has a deal with Valve allowing developers to export standalone games that ship on Steam with their own store page, royalty-free.

Source 2 Rendering

PBR materials, GPU ray-traced baked lighting (VRAD3), and the same visual quality as Half-Life: Alyx. Your games look stunning out of the box.

C# Scripting

Write all gameplay code in modern C# with hot-reload — change code, save, and see results instantly without restarting the game.

Built-in Multiplayer

Steam networking with NAT traversal, lobbies, and up to 64 players. Add [Sync] to a property and it replicates automatically.

Standalone Export

Ship your game on Steam with its own store page. Valve takes their standard Steam cut — no additional engine royalties.

Chapter 02

Installation & Setup

Getting started with S&Box takes about 5 minutes:

  1. Install S&Box from Steam (search "s&box" in the Steam store)
  2. Launch S&Box — it will download the latest engine and tools
  3. Install a code editor — Visual Studio Code, JetBrains Rider, or Visual Studio all work. Rider has the best C# experience, but VS Code is free and works great
  4. Create a new project — from the S&Box launcher, create a new project using the Starter template
Tip
When you first open a project, you'll see a minimal scene with just a Main Camera and Directional Light. This is intentional — S&Box gives you a blank canvas rather than a cluttered template.

Chapter 04

Core Concepts

GameObjects & Components

Everything in S&Box is built from two primitives:

  • GameObject — a container. It has a Transform (position, rotation, scale) and holds Components. On its own, it does nothing.
  • Component — the logic. A C# class that attaches to a GameObject and gives it behavior. A Model Renderer makes it visible. A Rigid Body makes it physical. Your custom code makes it do whatever you want.

You build game entities by combining Components on GameObjects. A player character might have: a CharacterController (physics), a SkinnedModelRenderer (visuals), a CombatController (your custom combat logic), and a HealthComponent (your custom health system). Each piece is independent and reusable.

Scenes

A Scene is a complete game level or environment. It contains a hierarchy of GameObjects. You set your Startup Scene in Project Settings — that's what loads when the game launches (typically your main menu).

You can load scenes at runtime with Scene.Load(), and you can have a System Scene that persists across scene changes (useful for game managers, audio systems, etc.).

Prefabs

A Prefab is a saved, reusable GameObject template. You build a GameObject with all its components and children, right-click it in the Hierarchy, and select "Create New Prefab". Now you can spawn copies of it anywhere — in the editor or from code at runtime.

Prefabs are essential for anything you spawn dynamically: player characters, projectiles, pickups, enemies, effects.

// Spawn a prefab from code
var instance = MyPrefab.Clone( spawnPosition );

The Component Lifecycle

When your component runs, the engine calls these methods in order:

MethodWhen It's CalledUse It For
OnAwake()Once, when the component is createdCache references, initialize state
OnStart()Once, after all components have awakenedLogic that depends on other components being ready
OnUpdate()Every frameInput handling, visual updates, UI
OnFixedUpdate()Every physics tick (50 Hz)Physics, movement, consistent simulation
OnEnabled()When the component is enabledRestart logic, subscribe to events
OnDisabled()When the component is disabledCleanup, unsubscribe from events
OnDestroy()When the component is destroyedFinal cleanup
Key Difference: OnUpdate vs OnFixedUpdate
OnUpdate() runs every frame — the rate varies with framerate (60fps = 60 calls/sec, 144fps = 144 calls/sec). Use it for input and visuals.

OnFixedUpdate() runs at a fixed rate (50 Hz by default) regardless of framerate. Use it for physics and movement so your game behaves consistently on all hardware.
Chapter 05

Your First Component

Here's where the real fun begins. Let's create a custom component from scratch.

Step 1: Create the File

Navigate to your project's Code folder in the Asset Browser (or in your file system). Create a new file called HelloWorld.cs.

Step 2: Write the Code

using Sandbox;

public sealed class HelloWorld : Component
{
    // This appears as an editable field in the Inspector
    [Property] public string Message { get; set; } = "Hello S&Box!";

    protected override void OnStart()
    {
        Log.Info( Message );
    }

    protected override void OnUpdate()
    {
        if ( Input.Pressed( "Jump" ) )
        {
            Log.Info( "I pressed jump!" );
        }
    }
}

Step 3: Add It to a GameObject

  1. Save the file — S&Box hot-reloads automatically
  2. Select any GameObject in the Hierarchy
  3. In the Inspector, click the green "+ Add Component" button
  4. Search for "HelloWorld" — your component appears!
  5. Click it to add it

Step 4: Press Play

Hit the Play button at the top of the viewport. Check the Console tab at the bottom — you should see your message printed. Press Space and watch it print again.

Screenshot: Your component working Add a screenshot showing the console output with "Hello S&Box!" and "I pressed jump!" messages
Understanding [Property]
The [Property] attribute is what makes a field appear in the Inspector. Without it, the field exists in code but you can't see or edit it in the editor. You can use [Property] on strings, floats, ints, bools, Vector3s, GameObjects, Components, Models, and more.

A More Useful Example

Here's a component that actually does something visible — it spins a GameObject:

using Sandbox;

public sealed class Spinner : Component
{
    [Property] public float Speed { get; set; } = 90f;

    protected override void OnUpdate()
    {
        Transform.LocalRotation *= Rotation.FromYaw( Speed * Time.Delta );
    }
}

Add this to a Cube, press Play, and watch it spin. Change the Speed value in the Inspector while the game is running — it updates in real time.

Reading Input in Code

protected override void OnUpdate()
{
    // True on the exact frame the key is pressed
    if ( Input.Pressed( "Jump" ) )
        DoJump();

    // True every frame while the key is held down
    if ( Input.Down( "Run" ) )
        speed *= 1.5f;

    // True on the exact frame the key is released
    if ( Input.Released( "PrimaryAttack" ) )
        FireProjectile();

    // Analog input (movement)
    Vector3 move = Input.AnalogMove; // X = right, Y = forward
}

Adding Custom Actions

Open Project Settings → Input and click "Add New Action..." at the bottom. Give it a name (e.g., "Dodge"), assign a key binding, and you can immediately use Input.Pressed("Dodge") in your code.

Chapter 07

Physics & Movement

Two Controller Types

S&Box offers two built-in ways to make a character move:

Player Controller

The all-in-one solution. Includes movement, camera, animation, footsteps, and the Citizen model. Drop it in and press Play — you're walking around immediately.

Uses a modular MoveMode system — Walk (priority 0), Ladder (5), and Swim (10) come pre-installed. Higher priority modes override lower ones automatically.

Best for: Rapid prototyping, games using the Citizen model, learning the engine.

Character Controller

The low-level physics capsule. You write all movement, camera, and animation code yourself. More work, total control.

Found under Add Component → Physics → Character Controller.

Best for: Custom characters, precise control over movement feel, games with unique movement mechanics.

Basic CharacterController Movement

using Sandbox;

public sealed class PlayerMovement : Component
{
    [Property] public float WalkSpeed { get; set; } = 200f;
    [Property] public float JumpForce { get; set; } = 350f;

    CharacterController cc;

    protected override void OnAwake()
    {
        cc = Components.Get<CharacterController>();
    }

    protected override void OnFixedUpdate()
    {
        // Build wish velocity from input
        var input = Input.AnalogMove;
        var wishVel = new Vector3( input.x, input.y, 0 ) * WalkSpeed;

        // Rotate movement to match camera direction
        wishVel *= Rotation.FromYaw( Camera.Rotation.Yaw() );

        cc.Accelerate( wishVel );
        cc.ApplyFriction( 5f );

        // Jump
        if ( Input.Pressed( "Jump" ) && cc.IsOnGround )
            cc.Punch( Vector3.Up * JumpForce );

        // Gravity
        if ( !cc.IsOnGround )
            cc.Velocity += Scene.PhysicsWorld.Gravity * Time.Delta;

        cc.Move();
    }
}

Chapter 08

Networking Basics

S&Box networking is built around a host-authoritative model using Steam's networking stack. One player (or a dedicated server) is the host — they run the authoritative game simulation. All other players are clients.

Key Concepts

ConceptWhat It Does
NetworkMode.ObjectMakes a GameObject networked — it can have an owner, synced properties, and RPCs
[Sync]A property that automatically replicates from the owner to all other clients
[Rpc.Broadcast]A method that runs on ALL clients when called
[Rpc.Host]A method that runs ONLY on the host — use for game-critical actions
[Rpc.Owner]A method that runs ONLY on the object's owner
IsProxyReturns true if you DON'T own this object — check this before writing state

Minimal Networked Component

public sealed class NetworkedHealth : Component
{
    // Automatically synced to all clients
    [Sync] public float Health { get; set; } = 100f;

    protected override void OnUpdate()
    {
        // Only the owner can modify their own health
        if ( IsProxy ) return;

        if ( Input.Pressed( "PrimaryAttack" ) )
            RequestDamage( 10f );
    }

    // Runs on the host — validates and applies damage
    [Rpc.Host]
    void RequestDamage( float amount )
    {
        Health -= amount;
        BroadcastHitEffect();
    }

    // Runs on everyone — plays visual/audio effects
    [Rpc.Broadcast]
    void BroadcastHitEffect()
    {
        Sound.FromWorld( "hit_sound", WorldPosition );
    }
}
Golden Rule for Multiplayer
Never trust the client. For any game-critical action (damage, scoring, item pickups), have the client REQUEST the action via [Rpc.Host], validate it on the host, then broadcast the result. This prevents cheating.
Chapter 09

Custom Assets Pipeline

If you want your game to look unique, you need custom models, textures, and animations. S&Box uses a Blender → FBX → ModelDoc pipeline to bring assets into the engine.

The Pipeline Overview

  1. Model in Blender — create your mesh, UV unwrap, texture, rig with an armature
  2. Export as FBX — use specific settings for Source 2 compatibility
  3. Import into S&Box — drop the FBX into your project's Assets folder
  4. Set up in ModelDoc — open the Model Editor, configure materials, LODs, hitboxes, physics, and animation sequences
  5. Assign an AnimGraph — connect an animation graph to drive the model's animations

Blender FBX Export Settings

ModelDoc

ModelDoc is the Model Editor — open it from Tools → Model Editor. This is where you configure everything about your model after importing the FBX.

The ModelDoc outliner shows the model's structure:

  • BodyGroupList — body part variants (different heads, armor pieces)
  • LODGroupList — Level of Detail meshes for different distances
  • MaterialGroupList — material/skin variants
  • RenderMeshList — the actual mesh data
  • AnimationList — animation sequences imported from FBX files
  • HitboxSetList — hitbox volumes for damage detection
  • PhysicsShapeList — physics collision shapes
  • AttachmentList — attachment points (weapon sockets, effect origins)

Textures & Materials

S&Box uses PBR materials (Physically Based Rendering). For each model you typically need:

Texture Rules
Always use power-of-two dimensions (256, 512, 1024, 2048, 4096). Maximum texture size is 4096×4096. Use the smallest size that looks good — fewer pixels means faster loading and less VRAM usage.

Chapter 10

AnimGraph Deep Dive

AnimGraph is Source 2's animation state machine system — and it's the most complex subsystem you'll encounter. It's also the most powerful.

The Key Paradigm Shift

AnimGraph Is Not Like Unreal
In Unreal, you call PlayMontage("Attack") from code. In Source 2, code never directly tells the animation system what to play. Instead, your code sets parameters (speed, grounded, attacking), and the AnimGraph's internal logic decides what to play and how to blend. This decouples animation from code — powerful but requires a different mindset.

How It Works

Every frame, the AnimGraph evaluates like a tree:

  1. Your C# code sets parameter values (move_speed = 150, b_grounded = true)
  2. The graph evaluates from leaves to root
  3. Each node blends, selects, or transforms animation poses
  4. The final pose reaches the Final Pose root node and gets rendered

Node Types

Node TypeWhat It Does
SequencePlays a single animation clip
Blend Space 1D/2DBlends between animations based on a parameter (e.g., speed blends walk→run)
State MachineSwitches between states with defined transitions (idle→walk→run→jump)
Layered BlendOverlays one animation on another (upper body combat over lower body locomotion)
AdditiveAdds an animation on top of another (breathing, aim offset, hit reactions)
SelectorPicks one of several inputs based on a parameter (weapon type selects hold pose)
IK NodesInverse kinematics — feet placement, hand attachment, look-at targets
Direct PlaybackPlays a specific animation directly (for special cases like emotes)

Parameters — The Bridge Between Code and Animation

Parameters are the values your C# code sends to the AnimGraph. The Citizen model's AnimGraph uses these key parameters:

ParameterTypeWhat It Controls
move_x, move_y, move_zFloatMovement speed in each axis (absolute units/sec)
move_speedFloatTotal movement speed (direction-independent)
move_groundspeedFloatGround-plane speed (most commonly used)
duckFloat0 = standing, 1 = crouching, 2 = extra low
holdtypeIntWeapon hold type (pistol, rifle, melee, etc.)
holdtype_attackIntWhich attack animation to play
b_groundedBoolIs the character on the ground?
voiceFloatVoice chat volume (animates mouth)

Setting Parameters from Code

// Using CitizenAnimationHelper (easy way)
var helper = Components.Get<CitizenAnimationHelper>();
helper.WithVelocity( cc.Velocity );
helper.IsGrounded = cc.IsOnGround;
helper.DuckLevel = Input.Down( "Duck" ) ? 1f : 0f;

// Direct parameter setting (full control)
var renderer = Components.Get<SkinnedModelRenderer>();
renderer.Set( "move_groundspeed", velocity.Length );
renderer.Set( "b_grounded", isGrounded );
renderer.Set( "holdtype", 5 ); // melee_punch

The Citizen's AnimGraph Structure

The built-in Citizen AnimGraph is organized into 5 sections — studying it is the best way to learn:

  • Section 01 — Central poses and animation sources. The holdtype selector that routes to weapon-specific branches.
  • Section 02 — Core locomotion. 2D Blendspaces driven by root motion speed values, ducking, and skidding.
  • Section 03 — Other movement states: jumping, landing, airborne, height scaling, AnimTag assignment.
  • Section 04 — Weapon branches (holdtypes). Each weapon type has its own blend pipeline with IK management, attack additives, and handedness support.
  • Section 05 — Final compositing: Direct Playback → Clothes → Foot IK → Final Pose output.

Three Approaches to AnimGraph

1. Extend the Citizen Graph

Add your own weapon holdtypes and states to the existing Citizen AnimGraph. Fastest path, but you're tied to the Citizen skeleton and conventions.

2. Fork the Citizen Graph

Copy the Citizen's AnimGraph and modify it heavily. You keep the proven locomotion system but can reshape everything else.

3. Build from Scratch

Create a completely new AnimGraph for your custom skeleton. Most work, total control. Required if your character isn't humanoid.

Chapter 11

Razor UI System

S&Box uses Razor for UI — if you know HTML and CSS, you already know 80% of it. Razor panels are essentially web pages rendered inside your game.

Two Types of UI

Screen UI (HUD)

Renders on top of the game view, like a HUD overlay. Health bars, minimaps, crosshairs, menus. Created by adding a UI → Screen GameObject.

World UI

Renders in 3D space — floating health bars above enemies, in-world signs, interactive terminals. Created with UI → World GameObject.

Creating a Simple HUD

Create a .razor file in your project's Code folder:

<!-- HealthHud.razor -->
@using Sandbox
@using Sandbox.UI
@inherits PanelComponent

<root>
    <div class="health-bar">
        <div class="health-fill"
             style="width: @(HealthPercent)%"></div>
    </div>
    <div class="health-text">@Health / @MaxHealth</div>
</root>

@code {
    float Health => Components.Get<HealthComponent>()?.CurrentHealth ?? 0;
    float MaxHealth => Components.Get<HealthComponent>()?.MaxHealth ?? 100;
    float HealthPercent => (Health / MaxHealth) * 100f;
}

<style>
    .health-bar {
        width: 300px;
        height: 20px;
        background: rgba(0, 0, 0, 0.6);
        border-radius: 4px;
        overflow: hidden;
    }
    .health-fill {
        height: 100%;
        background: linear-gradient(90deg, #ff4444, #44ff44);
        transition: width 0.3s ease;
    }
    .health-text {
        color: white;
        font-size: 16px;
        margin-top: 4px;
    }
</style>
Razor = HTML + C# + Scoped CSS
Each .razor file contains three sections: the HTML template at the top, a @code block with C# logic, and a <style> block with CSS that's automatically scoped to this panel only. If you've ever used Vue.js single-file components, it's the same concept.
Chapter 12

Building Maps with Hammer

Hammer is Source 2's built-in level editor — the same tool used to build Counter-Strike 2 and Half-Life: Alyx maps. Open it from Tools → Hammer.

Hammer vs Scene Editor

S&Box has two ways to build environments:

Scene Editor

Place GameObjects, configure components, set up gameplay logic. Best for dynamic game content, prefab placement, and scripted elements.

Hammer Editor

Build level geometry, sculpt terrain, place lights, bake lighting. Best for static environments, architectural spaces, and polished level design.

Most games use both together: build your map geometry in Hammer, then load it into a Scene using a MapInstance component, and add gameplay objects in the Scene Editor.

Hammer Editor Layout

  • Top toolbar — editing modes: Vertices, Edges, Faces, Meshes, Objects
  • Left toolbar — modeling and placement tools
  • Center — 3D Map View for editing
  • Right — Outliner (object list) and Object Properties
  • Bottom — Asset Browser with Cloud Browser for community assets

Lighting in Hammer — Your Biggest Visual Advantage

Source 2's baked lighting system is the same one that made Half-Life: Alyx look extraordinary. This is genuinely one of the engine's strongest features.

Light Types

LightDirectIndirectShadowsCost
Baked (Static)Baked into lightmapsBaked + light probesContact hardening, high qualityFree at runtime
StationaryDynamicBakedDynamic possibleModerate
DynamicFully dynamicNoneFully dynamicExpensive
Lighting Strategy
Bake all environment lighting for stunning quality at zero runtime cost. Reserve dynamic lights for gameplay effects only (muzzle flash, explosions, player flashlight). Limit to 1-2 baked lights per surface — more than 4 causes artifacts. Use Hammer's "Baked Lighting Complexity" visualization to check.

VRAD3 — GPU Ray-Traced Light Baking

VRAD3 uses your GPU's ray tracing hardware (RTX or RDNA2+) to bake lightmaps. This produces global illumination with realistic light bounces and contact-hardening shadows — the same quality as offline rendering, baked into your map for free at runtime.

Player Spawning Pattern

public sealed class GameNetworkManager : Component, Component.INetworkListener
{
    [Property] public GameObject PlayerPrefab { get; set; }

    // Called on the HOST when a player fully connects
    public void OnActive( Connection connection )
    {
        var player = PlayerPrefab.Clone( GetSpawnPoint() );
        player.NetworkSpawn( connection );
    }
}

Sync Properties — Deep Dive

AttributePurpose
[Sync]Replicates from owner to all clients
[Sync(SyncFlags.FromHost)]Only the host can write; all clients receive read-only
[Sync(SyncFlags.Query)]Polls the getter every tick instead of relying on setter detection
[Sync(SyncFlags.Interpolate)]Smoothly interpolates the value on remote clients

RPC Filtering

// Send to specific connection only
using ( Rpc.FilterInclude( targetConnection ) )
    SendPrivateMessage( "Only you can see this" );

// Send to everyone except one player
using ( Rpc.FilterExclude( localConnection ) )
    PlayThirdPersonAnimation();

// Send to connections matching a condition
using ( Rpc.FilterInclude( c => c.Ping < 200 ) )
    SendHighPriorityEvent();

NetworkOrphaned — What Happens When Players Disconnect

PolicyBehavior
DestroyObject is destroyed (good for player characters)
ClearOwnerOwner removed, host takes over
HostOwnership transfers to host
RandomRandom remaining player gets ownership

Dedicated Servers

For competitive games, dedicated servers eliminate the "host advantage" (0ms ping):

  1. Install SteamCMD
  2. Run: ./steamcmd +login anonymous +app_update 1892930 validate +quit
  3. Create a launch script: sbox-server.exe +game yourname.yourgame mapname +hostname "Server Name"
Security for Competitive Games
In Project Settings → Networking, uncheck all Default Client Permissions (Client Object Spawning, Refreshing, Destroying). These are enabled by default for convenience, but for PvP games they allow clients to manipulate networked objects — a major exploit vector.
Chapter 14

Performance & Optimization

Making your game run well is just as important as making it fun. Here's what matters in Source 2.

The Rendering Pipeline

Source 2 can comfortably render 1-3 million triangles on screen on mid-range hardware. But raw poly count is rarely the bottleneck — draw calls are. Every unique material on screen is a separate draw call. Reducing material count matters more than reducing triangle count.

Culling — How the Engine Avoids Rendering What You Can't See

SystemWhat It DoesYour Role
Frustum CullingHides objects outside camera viewAutomatic — no action needed
Occlusion CullingHides objects behind solid geometryAutomatic — works best in indoor maps
Octree PartitioningSpatial indexing for fast visibility checksAutomatic — replaced BSP in Source 2
Fade DistanceProps stop rendering beyond a set distanceSet per ModelRenderer — use aggressively!
LOD SystemSwaps to lower-poly meshes at distanceSet up LOD groups in ModelDoc
Area PortalsDoors/windows that block visibilityPlace in Hammer for indoor maps

Server Scalability

Player CountFeasibilityRecommendation
2 (1v1 duel)TrivialNo optimization needed
10 (5v5 teams)EasyDefault settings work fine
16 (FFA)ComfortableWatch particle/effect counts
32AchievableMinimize [Sync] properties, use LODs
64ChallengingLower tick rate, careful bandwidth management

Optimization Checklist

Models

Use LODs for every prop. Set fade distances on ModelRenderer. Minimize material count. Bake high-poly detail into normal maps.

Textures

Use power-of-two sizes. Reserve 4096 for hero assets only. Atlas small props onto shared texture sheets.

Lighting

Bake everything static. 1-2 baked lights per surface max. Reserve dynamic lights for gameplay effects only.

Networking

Only network what's needed. Use Unreliable RPCs for cosmetic effects. Lower tick rate for larger player counts.

Code

Cache Scene.GetAll() results. Use OnFixedUpdate for physics, OnUpdate for visuals. Check IsProxy early to skip unnecessary work.

Scenes

Use Terrain for outdoor areas. Aggressive fade distances on background props. Area portals for indoor maps.

  • Input Debugger — shows which inputs are active
Final Advice
Start small. Make a spinning cube. Then make it respond to input. Then add physics. Then add a second player. Layer complexity one piece at a time. Every game ever shipped was built incrementally — yours will be too.
